## Overview

This repository follows **Clean Architecture / Domain-Driven Design** principles with a clear separation of concerns across three layers:

1. **Handler Layer** (`internal/api/{domain}/`) - HTTP request/response handling
2. **Service Layer** (`internal/{domain}/`) - Business logic
3. **Repository Layer** (`internal/{domain}/storage/`) - Database access

When working on this codebase, always respect these boundaries and understand that:
- Handlers depend on services (never directly on repositories)
- Services depend on repositories and other services
- Repositories depend only on the database
- Domain models are used in services, entity models are used in repositories

## Understanding the Architecture

### Layer Responsibilities

**Handler Layer:**
- Deserializes JSON requests into DTOs
- Maps DTOs to domain models
- Calls service methods with domain models
- Maps service results to response DTOs
- Maps service errors to HTTP status codes and user-friendly messages

**Service Layer:**
- Contains business logic
- Validates domain models beyond basic request validation
- Maps between domain models and entity objects
- Coordinates between multiple repositories/services when needed
- Uses domain models in method signatures

**Repository Layer:**
- Provides database access
- Works with entity objects (generated by sqlboiler)
- Returns entity objects to services
- Located in `internal/{domain}/storage/`

### Domain Structure

Each domain (e.g., `interaction`, `profile`, `safety`, `conversation`) follows this structure:
```
internal/{domain}/
├── domain/
│   └── domain.go          # Domain models
├── mapper/
│   ├── domain_to_entity.go
│   └── entity_to_domain.go
├── storage/
│   └── repository.go      # Repository interface and implementation
└── service.go             # Service interface and implementation
```

The API layer mirrors this:
```
internal/api/{domain}/
├── dto/
│   ├── request.go
│   ├── response.go
│   └── mapper/
│       ├── request_to_domain.go
│       └── domain_to_response.go
└── handler.go
```

## Working with the Codebase

### Before Making Changes

1. **Read relevant files**: Understand existing patterns in similar domains
2. **Check `internal/README.MD`**: Review the detailed examples and patterns
3. **Understand the domain**: Identify which domain your changes affect
4. **Check dependencies**: Understand which services/repositories are involved

### When Creating New Features

Follow the exact steps outlined in `internal/README.MD`:

1. **Create database migrations** (if needed)
   - Use `make migrate-create` to generate migration files
   - Write SQL in the migration file
   - Run `make migrate-up` then `make entity` to generate entity models

2. **Create scaffolding for all layers**
   - Handler: `internal/api/{domain}/handler.go` and DTOs
   - Service: `internal/{domain}/service.go`
   - Repository: `internal/{domain}/storage/repository.go`
   - Domain models: `internal/{domain}/domain/domain.go`
   - Mappers: Create mapper functions for conversions

3. **Implement from bottom up**
   - Start with repository methods
   - Then service methods (using domain models)
   - Finally handler methods (using DTOs)

4. **Register in router and main.go**
   - Add handler registration in `internal/http/router/router.go`
   - Add service/repository instantiation in `cmd/main.go`

### Code Patterns to Follow

**Handler Pattern:**
```golang
func (h *handler) MethodName() http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        ctx := r.Context()
        
        // Extract user ID from context
        userID, ok := commoncontext.UserIDFromContext(ctx)
        if !ok {
            render.UnauthorizedResponse(w, r, h.logger)
            return
        }
        
        // Decode and validate request
        var req dto.RequestType
        if err := request.DecodeAndValidate(r.Body, &req); err != nil {
            h.logger.Sugar().Warnf("failed to decode request: %s", err.Error())
            render.Json(w, http.StatusBadRequest, commonMappers.ToSimpleErrorResponse("invalid request payload"))
            return
        }
        
        // Map request to domain model
        domainModel := dtoMapper.RequestToDomain(req, userID)
        
        // Call service method
        result, err := h.service.MethodName(ctx, domainModel)
        if err != nil {
            render.HandleServiceErrorResponse(h.logger, w, r, "MethodName", err, mapErrorsToStatusCodeAndUserFriendlyMessages)
            return
        }
        
        // Map domain model to response
        render.Json(w, http.StatusOK, dtoMapper.DomainToResponse(result))
    }
}
```

**Service Pattern:**
```golang
func (s *service) MethodName(ctx context.Context, domainModel domain.SomeModel) (domain.Result, error) {
    // Validate domain model
    if err := s.validateDomainModel(domainModel); err != nil {
        return nil, fmt.Errorf("validate domain model: %w", err)
    }
    
    // Map domain to entity
    entity := mapper.DomainToEntity(domainModel)
    
    // Call repository
    result, err := s.repo.SomeMethod(ctx, entity)
    if err != nil {
        return nil, fmt.Errorf("repository operation: %w", err)
    }
    
    // Map entity back to domain
    return mapper.EntityToDomain(result), nil
}
```

**Repository Pattern:**
```golang
func (r *repository) SomeMethod(ctx context.Context, entity *entity.SomeEntity) (*entity.SomeEntity, error) {
    result, err := entity.SomeEntities(
        entity.SomeEntityWhere.Field.EQ(value),
    ).One(ctx, r.db)
    
    if err != nil {
        return nil, err
    }
    
    return result, nil
}
```

## Common Tasks

### Adding a New Endpoint

1. Create request/response DTOs in `internal/api/{domain}/dto/`
2. Create mapper functions in `internal/api/{domain}/dto/mapper/`
3. Add handler method in `internal/api/{domain}/handler.go`
4. Add service method in `internal/{domain}/service.go`
5. Add repository method in `internal/{domain}/storage/repository.go`
6. Register route in `internal/http/router/router.go`
7. Ensure service is instantiated in `cmd/main.go`

### Modifying Database Schema

1. Create migration using `make migrate-create`
2. Write SQL in migration file (Up and Down sections)
3. Run `make migrate-up`
4. Run `make entity` to regenerate entity models
5. Update repository methods if needed
6. Update service/domain models if schema changes affect business logic

### Adding Error Handling

1. Define error variables in service layer:
   ```golang
   var (
       ErrSomeError = errors.New("some error message")
   )
   ```
2. Return errors from service methods with context:
   ```golang
   return fmt.Errorf("operation failed: %w", ErrSomeError)
   ```
3. Map errors in handler's `mapErrorsToStatusCodeAndUserFriendlyMessages`:
   ```golang
   case errors.Is(err, service.ErrSomeError):
       return http.StatusBadRequest, "User-friendly error message"
   ```

## Code Generation Patterns

### Mapper Functions

**Request to Domain:**
```golang
// internal/api/{domain}/dto/mapper/request_to_domain.go
func RequestToDomain(req dto.Request, userID string) domain.SomeModel {
    return domain.SomeModel{
        UserID: userID,
        Field:  req.Field,
        // ... map all fields
    }
}
```

**Domain to Response:**
```golang
// internal/api/{domain}/dto/mapper/domain_to_response.go
func DomainToResponse(domain domain.SomeModel) dto.Response {
    return dto.Response{
        Field: domain.Field,
        // ... map all fields
    }
}
```

**Domain to Entity:**
```golang
// internal/{domain}/mapper/domain_to_entity.go
func DomainToEntity(d domain.SomeModel) entity.SomeEntity {
    return entity.SomeEntity{
        Field: d.Field,
        // ... map all fields
    }
}
```

**Entity to Domain:**
```golang
// internal/{domain}/mapper/entity_to_domain.go
func EntityToDomain(e *entity.SomeEntity) domain.SomeModel {
    return domain.SomeModel{
        Field: e.Field,
        // ... map all fields
    }
}
```

## Best Practices

1. **Always use domain models in service interfaces** - Never expose entity models in service method signatures

2. **Keep handlers thin** - Handlers should only handle HTTP concerns (serialization, status codes), not business logic

3. **Validate in layers** - Use `validator/v10` tags in DTOs for basic validation, add complex validation in service layer

4. **Wrap errors with context** - Always add context when returning errors: `fmt.Errorf("operation: %w", err)`

5. **Use consistent naming** - Follow existing patterns in the codebase for naming conventions

6. **Check for existing patterns** - Before creating something new, check if similar functionality exists elsewhere

7. **Respect the dependency direction** - Handlers → Services → Repositories → Database

8. **Use transactions when needed** - For operations that need atomicity, use the Unit of Work pattern (UoW) as seen in the codebase

9. **Log appropriately** - Use structured logging with zap, log errors in handlers, log important state changes in services

10. **Update error mappings** - Always update `mapErrorsToStatusCodeAndUserFriendlyMessages` in handlers when adding new error types

## Key Reminders

- **Entity objects** are generated by sqlboiler and live in `internal/entity/` - don't modify them manually
- **Domain models** represent business concepts and live in `internal/{domain}/domain/`
- **DTOs** are for API contracts and live in `internal/api/{domain}/dto/`
- **Common code** should go in `pkg/commonlibrary/`
- **Migrations** use goose and are in `migrations/`
- **Always run** `make lint` and `make build` before considering work complete
- **Use the Makefile** for common operations - it standardizes commands across the team

## When in Doubt

1. Check `internal/README.MD` for detailed examples
2. Look at similar features in the codebase (e.g., if working on blocking, check how other safety features work)
3. Follow the existing patterns - consistency is key
4. When creating new domains, use the scaffolding pattern from `internal/README.MD` section "Creating a new feature"
